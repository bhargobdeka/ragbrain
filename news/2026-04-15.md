🧠 Learn & Understand

• State-Aware Retrieval in RAG — The field is moving away from static top-k retrieval toward systems that decide what to fetch based on current agent state (session history, prior retrieval, task stage). This is the core gap between toy RAG demos and production. Reddit thread this week is solid on this. [r/AI_Agents post]
• Composable Multi-Agent RAG Architecture — The pattern: Query Understanding → Retrieval Planning → Multi-Retriever Execution → Fusion → Reasoning, each as a swappable component. As basic RAG gets commoditized into vanilla LLM services, differentiation now lives in this orchestration layer — which is exactly your lane.
• Memento-Skills: Skills as first-class mutable units — Worth understanding the "skill as rewritable memory" abstraction. It's a concrete answer to the question "how do agents get better without retraining?" and the unit-test-gated mutation idea is production-safe.

───

🛠️ Build & Practice

• Implement a state-aware retrieval router — Build a thin layer that inspects agent state (current task step, prior retrieval results, conversation turn) before deciding retrieval strategy (skip, top-k, full re-rank, or query decomposition). Even a simple rule-based router shows the concept clearly and is immediately production-applicable.
• Prototype a skill-rewriting loop (lightweight Memento-Skills) — Store skills as versioned text files, have the LLM critique and patch them after failures, run a quick eval gate before promoting. Even a toy version of this on a side project will make the architecture click.

───

My take this week: The gap between "AI that uses tools" and "AI that improves its own tools" is closing fast — Memento-Skills is the clearest production-ready signal of that shift, and if you're building agents, understanding that pattern now puts you ahead of most teams who are still debating which vector DB to use.
